O Problema Não É Só Falta de 
Material
Excesso parado
Muitas famílias possuem livros e apostilas parados em casa, sem uso ao longo 
do ano letivo.
Demanda real
Outras famílias precisam desses materiais ao longo do ano letivo, mas não 
sabem onde encontrá-los.
Doação informal
A doação costuma acontecer de forma informal, espalhada e sem 
acompanhamento ou registro.
Falta de estrutura
Faltam busca, filtros, controle de pedidos e um fluxo mínimo de segurança para 
conectar doadores e responsáveis.
O EcoBook nasce para organizar esse ciclo do início ao fim.
Nossa Proposta: Um Fluxo Completo
Não é só um catálogo de livros, é um processo completo de publicar, buscar, solicitar e concluir a doação.
App Android Nativo
Voltado a pais, mães e responsáveis legais. 100% Kotlin e Jetpack 
Compose.
Backend Próprio
Autentica, valida regras de negócio e persiste os dados com API REST e 
banco relacional.
IA Assistiva
Gemini apoia o cadastro de materiais sugerindo metadados, sem tirar a 
revisão humana.
Notificações
FCM mantém doador e responsável informados ao longo de todo o fluxo 
de pedidos.

Stack Tecnológico
Android — Kotlin + Jetpack Compose
Kotlin + Compose: UI declarativa, 100% nativo, sem XML
ViewModel + StateFlow + Coroutines: estado reativo e 
assincronismo
Hilt: injeção de dependência em repositórios e serviços
Retrofit + OkHttp + Gson: consumo da API REST
EncryptedSharedPreferences: JWT, sessão, token FCM e inbox 
local
Coil + Timber: imagens e logging
FCM: push notifications
Backend — Spring Boot + PostgreSQL
Spring Boot + Security + JWT: API REST stateless para mobile
Spring Data JPA + Hibernate + Flyway: persistência relacional com 
migrações versionadas
PostgreSQL: banco principal; H2 para local/testes
Gemini + OkHttp: preview IA no endpoint /materiais/preview
Firebase Admin SDK: envio de push ao Android
Micrometer + Prometheus + OpenAPI: observabilidade e 
documentação
Fluxo técnico: Android → API JWT → services → PostgreSQL/arquivos → Gemini/FCM quando necessário.
Os Extras Sustentam Funcionalidades Reais
API Própria
Usada em autenticação, perfil, materiais, discovery, solicitações, 
notificações e consentimentos. Base de toda a persistência do 
produto.
Integração Externa, Gemini
Usada no preview do material para sugerir metadados (título, autor, 
editora, disciplina) antes da publicação pelo usuário.
Firebase Cloud Messaging
Push notifications para o fluxo de pedidos e aprovações, mantendo as 
duas partes informadas fora do app.

Onboarding 
Login e Onboarding
Autenticação com e-mail + senha no backend próprio (sem 
Firebase Auth). No primeiro acesso, o app direciona ao 
onboarding: CPF, WhatsApp, cidade, bairro e aceite dos termos. 
Fluxos protegidos só são liberados com responsável identificado.
Tela de Perfil
Login e Onboarding
Os dados do usuário cadastrado incluem: 
Email, senha, Whatsapp, CPF, cidade e bairro. Os dados de 
instituição de ensino, foto de perfil e tema escuro são 
complementares.
Todo usuário cadastrado deve previamente aceitar os termos do 
aplicativos e autorizar a classificação de materiais assistida por IA
Publicação com IA
Publicação de Material
Capa frontal obrigatória e traseira opcional. Android valida JPEG/PNG, comprime e 
envia upload. Backend cria load temporário e chama o Gemini (se houver 
consentimento de IA) para sugerir título, autor, editora, disciplina, nível, ano e sistema. 
O usuário revisa tudo manualmente antes de publicar.
Mesmo sem IA, o cadastro continua funcionando (a IA ajuda, mas não 
bloqueia o produto).
Publicação com IA
 Uso da câmera
A imagem apresentada pelo usuário doador, pode ser selecionada através da 
galeria de fotos, ou usando a própria câmera do celular
Tela de descoberta
🔍 Buscar
Cards com título, tags, local e estado do material. Busca com filtros acadêmicos e 
contexto geográfico por cidade e bairro. O detalhe abre imagens, descrição, ano e 
sistema de ensino antes do pedido.
Tela de solicitação
📋 Solicitações
Estado vazio orienta o usuário antes da primeira solicitação. Depois da aprovação, 
o card mostra status, reserva ativa e liberação de contato. Diferença clara entre 
pendente, aprovada, cancelada e concluída.
Solicitação e Acompanhamento
📬 Notificações
Doador enxerga pedidos pendentes e aprovados em áreas separadas. Pode 
concluir entrega ou revogar reserva. FCM envia push quando configurado; app 
guarda inbox local para não depender só do push do sistema.
Fluxo do usuário (Solicitante)
 Pedir material
Solicitante busca na aba de descobertas por materiais que necessite, com a ajuda 
de filtros como: Título, autor, sistema de ensino, localização, disciplina, nível de 
ensino, ano escolar e necessidade acadêmica
Fluxo do usuário (Doador)
 Aceitar Pedido
Doador é notificado, enxerga seus pedidos pendentes e avalia sua aprovação ou 
recusa do pedido
Fluxo do usuario (Solicitante)
 Falar com doador
Ao ter um pedido aprovado, o solicitante tem a oportunidade de falar com o 
doador através do Whatsapp para combinar as questões da entrega do material
Fluxo do usuário (Doador)
 Confirmar doação  
Após a conversa particular, o doador tem a opção de marcar a doação como 
concluída, ou revogar a aprovação anteriormente concedida
